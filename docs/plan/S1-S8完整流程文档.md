## 0. 数据纪律

1. recovery 结论必须来自 execute–then–undo trace；LLM 只能提案动作或改写措辞。
2. `RECOVERED` 是构造性、solver/预算/σ 相对的证据。
3. `NOT_RECOVERED_WITHIN_BUDGET` 是有限搜索失败，不是不可逆证明。
4. recoverability 与 normative risk 独立；恢复状态不等于撤回现实伤害。
5. 无 `probe_point_id/state_id/action_instance_id` 的记录不得进入正式训练。
6. 不覆盖原始 JSONL；legacy、mock、失败轨迹和开发数据必须显式隔离。

## 3. S1：日志、指纹、历史与 AXTree

### 3.1 不可变 collection run

新 collection attempt 使用不可变 `run_id` 和 physical trajectory identity。重复 logical task/seed 是独立 run，不能覆盖 raw 或假装成同一独立样本。active formal SFT 的62个views关联5个runs/19个trajectories与19个states，closure audit为0问题。全量meta与旧阶段计数只保留作历史审计，不能用全量meta行数代表独立成功轨迹数。失败轨迹没有混入active formal expert SFT。

### 3.2 统一 train/deploy 输入

唯一 serializer：`revact/prompts.py::build_policy_messages`。

```text
messages[0].role = system
messages[0].content = {agent_system}
messages[1].role = user
messages[1].content =
<goal>
{goal}

<history>
{history_block(last K observed-delta rows)}

<observation>
{current pruned AXTree}
```

`build_policy_messages` 只返回上述 `system + user` 输入；SFT 物化时才另附 assistant completion。user content 的最后一行带换行符。

以上与 `revact.prompts.render_user` 一致：`<goal>`、`<history>`、`<observation>` 是开标签分隔符，不写 closing tags。

- `POLICY_HISTORY_STEPS=9` 同时驱动 collection、IrisPolicy、rollout 和新 materializer；
- single/multiturn history flag 都由精确 raw trajectory 的相邻观测 diff 重算，不使用 grounded label，并通过 COMPLETE run manifest→meta→raw→source state closure；
- `no-effect` 表示可见状态未变化，不代表动作安全；
- action-anchored pruning 保留 supervised bid 和祖先上下文；
- 缺失 supervised bid 时 assembler/trainer 直接拒绝。

旧 multi SFT 中 risky bid 仅 12/62 可见；新输入 gate 不允许直接训练该文件。

当前 active 62 views 的formal-input诊断中历史长度全部为1；`stateless_episode_traces.v1.jsonl` 及manifest尚不存在，因而readiness中1–3/4–6/≥7的canonical episode history buckets全为0，中间state-changing supervision也为0。formal-input中的短history不能冒充episode-level证据。

## 4. S2：轨迹与关键状态发现

`afforded_action_types` 仍只覆盖 `add_to_cart/place_order/delete_address`，但新 collector 已把它降为 `keyword_ranking_hints`，不再作为发现门；候选状态来自全部 legal interactive controls。当前 `shopping_key_states.jsonl` 281行，其中 legacy 241行的 210/29/2 分布仍是旧 shallow-keyword 产物；`reddit_key_states.jsonl` 7行，`formal_point_reached_states.jsonl` 31行。三者是可重叠视图，后者还是派生索引，不会把旧分布变成任务无关 census。

代码中的 mutation detector 协议为：

1. reset 到可恢复 snapshot；
2. 枚举全部合法 interactive controls；
3. 安全执行候选；
4. 比较持久 UI/API/DB signal；
5. reset；
6. 保存 mutation candidate 和 detector provenance。

关键词、LLM 和 sitemap 只能排序/覆盖采样，不得产生 label。当前 release-scoped live mutation report/manifest 缺失，样本数为0；任何离线 fixture 数字都不计入门槛。正式上线前仍需在至少 200 个 live controls 上报告 mutation precision/recall 与 Wilson interval。

`reach.py/scale.py` 的稳定 URL 和运行时 bid 解析可用于创建候选状态。正式 point 还要求：

## 7. S5：point-level grounding

### 7.1 Canonical ontology

```text
effect_status:
  CHANGED | NO_EFFECT | UNKNOWN

recovery_status:
  RECOVERED | PARTIALLY_RECOVERED |
  NOT_RECOVERED_WITHIN_BUDGET | UNKNOWN

undo_cost_steps: nullable integer
```

新 formal body 永不写 `IRREVERSIBLE`。legacy reader 可映射该词，但不得提升证据强度。

### 7.2 Point admission

`GroundingPoint.validate(formal=True)` 要求完整 point identity、transition hashes、signals、solver set、budget 和 evidence。`save_probe_points` 同时维护 body/manifest 1:1 和 immutable ID。

普通 `save_results` 只写 `grounded/smoke/`，不会再追加冻结的 legacy body。正式路径为 `collect-points → prepare-point-batch → probe-points`：reviewed label-free execution spec 固定 state/candidate/action/snapshot，runner 保存 pre/post/final signals、observation hashes 与实际 undo trace。当前已产出32条point body+manifest；其中19条另保存完整transition body+manifest。余下13条不能从hash反推或重建observations，`revact probe --formal-spec`兼容入口也不能把class smoke升级成point。

### 7.3 Negative protocol

正式负点需要 deterministic controller + affordance BFS + LLM undo attacker，任一恢复即翻标签，并报告 attack break rate。当前32点均未产生 `NOT_RECOVERED_WITHIN_BUDGET`；formal negative solver-union evidence=0。LLM attacker prompt 已入 registry，但未形成符合门槛的live多路trace，BFS branch reset和API/DB signal也未达正式负证据要求，因此不得报“不可逆率”。

### 7.4 σ 的边界

当前32/32 points只有 `ui_structural` 信号，经验证的 API/DB signal assets=0。UI 信号不是数据库上界，可能漏掉邮件、支付、webhook、审计记录、缓存、异步状态和第三方观察。每个 action class 都必须发布 signal coverage 声明。

## 8. S6：decision truth

正式 truth 由独立轴组成：

- `effect_status/recovery_status`
- `normative_risk` 或 `policy_constraint_truth`
- `action_required_for_goal`
- `violates_constraint`
- `expected_decision`

正式 oracle 不从 recoverability 单独推出安全结论。历史固定 risk 数值只能用于 legacy compatibility，不能报 ECE。

## 9. S7：teacher 与 prompt provenance

正式 teacher 只能改写由 point evidence pin 住的事实。QC 检查：

- 标签结构和顺序；
- input observation evidence；
- measured post diff；
- undo action/cost；
- recovery、decision 和 answer consistency；
- `prediction_source=probe_transition`；
- `undo_source=probe_point_id`。

prompt 以 content-addressed bundle 保存全文、fingerprint 和父版本；另用 `prompt_generation_fp` 对 producer/model/decode 配置独立寻址。registry 现覆盖 agent、collector、teacher、undo attacker 和目标模板。active single/multiturn 62个 source views 已写入可解析 provenance；2026-07-16 active v5 teacher 为 single 38/38、multiturn 24/24 QC-passed rows，合计62/62。旧v3/v4与47/48阶段仍保留作历史QC资产，不再计入active coverage。历史 `prompts_fp=259fceac4e0c` 仍缺全文 bundle。

工作台已能读取、浏览并导出 single/multiturn SFT/DPO，也展示 candidate、formal/legacy grounding、truth、teacher 和 opinion。当前可展示32-point lineage、150个active candidates，以及19条完整`state → candidate → transition → probe → truth → teacher/sample`链和62个active teacher/sample views。opinion只覆盖6/64 truth cases且全为LLM rater，human=0。base split/export仍明确阻塞。OpenRouter 是本次执行配置，不是全局默认；前端仍由用户选 provider/base URL/model/key env。

## 10. S8：assembly、split 与训练

### 10.1 Formal assembly

正式样本必须通过唯一 `probe_point_id` join，且：

- input 使用共享 stateless builder；
- risky bid 可见；
- prediction/undo 引用 transition/point evidence；
- `is_mock=false`；
- expert trajectory 必须声明 `collector_success=true`，并与不可变 COMPLETE collection run 精确连接；当前62个active views的closure audit为0问题；
- assistant turn type 显式；
- tokenizer 超限 count=0。

当前正确active组装结果是 single 38 / multiturn 24 个 smoke views，且绝不回退 action-type label。两族合计62 views，只代表19个有完整transition body的points；其余13个hash-only points被明确排除。24行multiturn的 `history_steps_total=1`、`n_turns=1`，不支持长历史主张。

### 10.2 Split

`data/splits.py` 对 single 与 multiturn 做一次联合 connected-component 分配，以 state/entity/template/environment 为 publication isolation 轴，并额外报告 action/site/privilege overlap。train/dev/test 三侧必须非空；任何跨 family 重叠、旧 shard 残留或缺失 `SPLIT_REPORT` 都使 formal export fail closed。cross-action、cross-privilege、cross-template、cross-site 只有在至少两个非空组且 train/test 都非空时才标 `available=true`。

代码能生成 challenge split 不代表实验已经存在。`*_cross_<axis>*.jsonl` 路径可被
物化为 0 行 stale-clearing sentinel；只有 `SPLIT_REPORT` 标记 `available=true` 且两侧
非空才是可用 split。当前active base train/dev/test为0/0/0，19个transition-backed states在 state/entity/goal-template/page-template/environment 联合隔离图中仍无法分成三个非空且无泄漏的shards；formal export因此fail closed。旧v1 challenge shards保留为superseded诊断资产，不等于已有泛化结果。cross-privilege/cross-template也不可用。

### 10.3 训练定位

- SFT：completion-only，formal validator fail closed；
- DPO：synthetic 和 deployment negatives 分层；active v3有6个legal-candidate +4个verified on-policy pairs；旧single/multi各12对legal-candidate和旧v1/pre-fix supplements已历史化，base DPO train仍为0；
- GRPO/GSPO：固定 prompt 的 offline RLVR ablation，不是环境在回路的多轮 RL。

当前 teacher/body validation 已通过，但 strict base split 仍为空；本次未启动 GPU 训练、coupled/decoupled 实验或 environment-in-the-loop online RL。
