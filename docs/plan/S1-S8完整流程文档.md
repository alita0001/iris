## 0. 一条贯穿始终的铁律

**安全"结论"必须行为 grounded，"措辞"才可以模板/蒸馏。**

| 内容 | 来源 | 能否用 LLM |
|---|---|---|
| `reversibility` + `undo` 计划 + undo 代价 | **环境里真实 execute-then-undo 测量**（grounding 层）| ❌ |
| `constraint_violation` | 目标变体构造规则判定 | ❌ |
| `<prediction>` 的参照 | 真实 `O_{t+1}` | ❌ |
| `<rev_check>` 的**事实**（undo 通道存在性/实测步数/残差，P2） | 探针 `undo_actions/undo_steps/residual_diff` 实录 | ❌ |
| `<observation>/<reasoning>/<prediction>/<rev_check>` **措辞** | 模板 或 teacher **条件蒸馏**（结论 pin 死 + QC 矛盾拒绝，`revact/train/distill.py`）| ✅ 只写理由，不定结论 |
| 输入侧 `<history>` 行的 flag/delta（P2） | 相邻真实观测 fingerprint diff（`obs_delta`） | ❌ |
| `<answer>` | 专家动作 / 规则安全替代 | ❌ |

## 2. S1 —— 环境、指纹、复现

- `envs/harness.py`：`make_env`（懒加载 browsergym + `webarena_patch.configure_reward_judge()`）；`RevActEnv`（step 级日志 + replay 前缀）；`replay_to_state`。
- `envs/fingerprint.py`：指纹剥 `[bid]` 后哈希；`is_restored` 有 backend_state 时以后端为准。
- `envs/mock_env.py`：离线状态机，**页面标记复刻 Magento 措辞**（`link 'Remove item'`、9 位订单号 gridcell、wishlist/newsletter 页），因此 grounding 信号与探针协议可完全离线测试。
- `envs/webarena_patch.py`：reward judge 三模式（off/deepseek/openai），`WA_*` 变量补齐。

**⚠️ 诚实声明（修正旧版表述）**：S5 探针**不使用** `replay_to_state`（录制的 bid 前缀跨会话失效）。探针到达状态用**确定性重导航**（稳定 Magento URL + 运行时按文本重解析控件）+ **探测前基线信号校验**（如先验证购物车为空）。`replay_to_state` 的实际用途是 S2 轨迹态的复现校验与离线测试（`tests/test_offline_pipeline.py`）。

## 5. S5 —— ⭐ 可逆性探针注册表（核心资产）

### 5.1 架构

- `grounding/base.py`：`ProbeSpec` 注册表 + `run_probe` **双闸门 runner**——破坏性探针需 `--commit` **且** `REVACT_ALLOW_DESTRUCTIVE=1`，缺任一强制 dry-run；探针异常收容为 UNKNOWN+traceback，批量探测不中断。每条结果带 `probe_id/timestamp/commit_mode`，另写 `MANIFEST.jsonl`（含 controller_version）供溯源。
- **防覆盖 loader**：`load_reversibility` 每类取**最新非 UNKNOWN** 行——dry-run 永远不能覆盖 grounded 标签（旧版 append+last-wins 的 bug 已修，有回归测试）。
- `grounding/signals.py`：后端信号 = 结构化 UI 读数（**在引号内名字里匹配**，容忍 Magento 的  图标前缀；行数权威、侧栏空文本仅参考；checkbox 解析 `checked='true|false'` 两种序列化）。
- `grounding/undo.py`：undo 控制器**实录 `undo_actions`**（undo 代价是测出来的），处理确认模态（compare 移除弹 OK 框），`budget_exhausted` 显式标记。

### 5.2 探针清单（13 个）与 live 实测状态

| 探针 | 等级 | live 标签（2026-07-02） | 证据摘要 |

| shopping.wishlist_add | 非破坏 | **REVERSIBLE** | 回到基线；已知问题：侧栏控件使计数含常量偏移，标签逻辑（复原到基线）不受影响 |
| shopping.newsletter_subscribe | 非破坏 | **REVERSIBLE** | checked True→False→True，undo_steps=2 |

## 6. S6 —— oracle decision

`assemble.oracle(reversibility, violates, requested)` 不变（只吃 grounded 输入，risk 分数随可逆性分档）。规则表见 06 号文档 §5；有单测覆盖。

## 7. S7 —— 措辞：模板 + teacher 条件蒸馏

MVP 模板（`build_fields`）之上新增 `train/distill.py`：teacher（DeepSeek，key 只从 env 读）收到 **pin 死的 grounded 结论**（含实测 undo 路径）只写**四行**措辞（`<observation>/<reasoning>/<prediction>/<rev_check>`，P2 起）；QC 用矛盾正则拒绝违背 pinned 标签的输出（如 label=IRREVERSIBLE 却写 "can be undone"），重试 ≤3 次后丢弃。`<reversibility>/<undo>/<decision>/<answer>` 字节级不动（有单测）。蒸馏模板在 prompt registry（`teacher_distill`，工作台可编辑）。命令：`python -m revact.cli distill --limit 10`。

## 8. S8 —— SFT + DPO 组装

- 标签绑定仍按 risky 动作文本关键词（ACTION_KW 已扩到 8 类动作）。
- **DPO 四类偏好对**（旧版仅 2 类）：`false_safe`（安全结论翻转+执行）、`over_block`（无谓拒绝）、`goal_violation`（**世界模型正确但违约执行**——把约束遵从与可逆性认知解耦）、`wrong_reversibility`（标签翻转、决策随错误标签走——监督标签保真）。
- 当前产物：**92 SFT / 230 DPO**（46 状态 × 2 变体；DPO 平均 2.5 对/样本）；另有 **62 多轮 SFT / 155 多轮 DPO**（P1，见下）。
- **P0（2026-07）**：SFT/DPO 的 user 输入统一为 `<goal>/<history>/<observation>` 三段（`revact/prompts.py`，与 rollout 策略同一 builder）；此前代码漏掉 history、训练分布≠部署分布，已修。meta 增 `history_source`、`risky_raw_action`。
- **P1（多轮样本）**：`revact/data/multiturn.py`（CLI `assemble-multiturn`）把真实轨迹 + key states + grounded 标签组装为**多轮 chat 样本**（途中步 `<answer> 动作`、风险决策步完整 `<think>` 块，loss 计全部 assistant 轮）。
- **P2（2026-07-08，格式 `iris.v2`，详见 06 计划书 §5.1/§5.1.1/§5.5）**：① `<history>` 行升级为编号三要素"动作 + [flag] + 关键 delta"（flag=nav/state-change/update/no-effect，由 `envs.obs_utils.obs_delta` 从相邻真实观测算出，绝不来自 grounded 标签；`[no-effect]` 防"动作无效→输入重复→死循环"）；② assistant 增 `<rev_check>`（label 前的 undo 机制检查，事实由探针 `undo_actions/undo_steps/residual_diff` 钉死）与 `<undo>`（计划+实测步数）；③ prompt registry：agent/采集/蒸馏 prompt 与目标模板池全部工作台可编辑，meta 增 `undo_steps/format/prompts_fp`。
- 命令：`python -m revact.cli assemble && python -m revact.cli assemble-multiturn && python -m revact.cli split`。

## 9. 规模化与切分

`revact.cli crawl --cap 40` → `revact.cli scale --n-place-order 6`（全非破坏）。**按 action-type 复用标签仅是 pilot 权宜**——正式协议是逐 (state,action) 探测 + 类型恒定性抽验（06 文档 §6.2）。`splits.py` 按商品切 70/22 + DPO 过滤（防泄漏有单测），并在代码注释中明示：**商品级 split 是弱检验（checkout 页与商品无关），真正的概念检验是 cross-action-class split**。

## 10. 训练与评测（含诚实口径）

- **三条训练路径**（均 LoRA，qwen-vllm 环境 + GPU；epochs 3、lr 1e-4——旧配置 6 epoch/2e-4 对模板数据过拟合）：
  - `train`（SFT，单步或多轮；多轮全 assistant 轮计损 + 梯度检查点，max_len>4096 时 bs=1）；
  - `train-dpo`（TRL DPOTrainer，四类偏好对；`--adapter` 从 SFT 热启，ref=SFT 策略）；
  - `train-grpo [--gspo]`（GRPO/GSPO，`revact/train/grpo.py`）：reward **离线可验证**（format+1 / decision 对 oracle+2 / reversibility 对实测标签+2 / 约束违反−4），无 LLM judge、无环境在回路；`--adapter` 从 SFT 热启。**定位：grounded-reward ablation，非主线贡献（见 06 文档 §7 Stage 3）**——RL 的作用是闭合 SFT/DPO 触及不到的 rollout 分布偏移。
  - 各入口均支持 `--dry-run`（校验 schema/标签/角色序，不导入 torch）。
- `revact.cli eval --dry-run` 校验测试集并**警告 n<5 的 cell**。
- **`revact.cli eval-rollout`（守护式端到端 FSR 评测，2026-07 新增）**：训练后策略经 vLLM LoRA 部署（base/iris-sft 双臂），held-out 状态真实 rollout；破坏性动作类只记录声明、绝不执行（从探针注册表导出 guarded 集），add_to_cart 类执行后 empty_cart 清理。Pilot（n=11，方向性）：base FSR 0% 但退化全拒角、iris-SFT FSR 9.1% 且请求侧 10/11 正确、单步 decision 声明准确率 86%（对比 base 的 1/22 可解析）。详见 06 文档 §9.6。
- **旧版"14%→100%"结果的诚实注记**：base 4/22 产出格式 → 该对比主要测格式遵循；100% 发生在决策与 (action_type×变体) 完全共线的 4-cell 设置下，**不构成"学会可逆性概念"的证据**。eval 现同时报告 format-valid 子集口径；概念性证据以 cross-action-class held-out 的 Reversibility-F1（对照按钮文字 logistic 下界）为准。**单步评测测不出多步失败，rollout harness 补上这一维度。**

## 13. 现状、局限与下一步

**已验证**：全链路（含 4 个新探针的 live grounding、双闸门、防覆盖、undo 代价实录）跑通；离线 31 测试全绿。

**局限（诚实）**：谱系中段（PARTIALLY/REVERSIBLE_WITH_COST）与 admin 侧标签**尚未采集**（骨架已备，commit 需批准）；address_add 表单校验待修；wishlist 计数有常量偏移；决策数据仍共线（等动作类扩到 8+ 后由 cross-action-class split 解决）；沙盒自动履约限制"不可逆"外部效度（agent-relative 定义应对，见 06 §3/§13）。

**下一步**（=06 文档 W1–W4）：非破坏探针 ×30–80 状态点/类上量 → 破坏类批准后抽样 → 蒸馏全量 + 重训 → 实验 0（H1 探针）+ cross-action-class 判决。
